Elastic stack 을 이용한 통계 서비스 구축

이번에 Elastic stack 을 이용하여 통계 서비스를 구축하게 되었다.
공유 할 만한 부분들을 정리해서 순차적으로 내용을 정리해볼 생각이다.

목차

  1. 들어가며
  2. 어떤 도구를 이용할 것인가?
  3. 어떻게 할 것인가?
  4. 통계 데이터를 위한 요구 사항 정리
  5. [Elastic stack 시스템 구축]

들어가며

나는 회사에서 특정 서비스를 개발하고 있다.
아직 부족한 기능이 많은 서비스였지만, 0에서 부터 시작을 했고, 그것을 “내가” 주도 했기 때문에
매우 많은 애착을 가지고 있었다.
(그래서 수많은 요구사항에도 불구하고, 매우 긍정적으로 피드백 하고자 노력했다)

그러던 어느날 올것이 왔다. 통계 기능이 필요하다는 요구사항이 많아지고 있다는 것 이었다.
구글리틱스, 에이스터 등의 몇가지 통계/분석 서비스를 “연동”한적은 있지만,
자체적으로 “구축”을 해본 경험은 전무했던터라 선뜻 하겠다는 얘기 보다는
잘 할수 있을까? 라는 생각이 먼저 들긴 했었다.

어떤 도구를 이용할 것인가?

하지만 어차피 해야하는것. 뒤로 미룰순 없었다. 부족한 경험으로 인해 정확한 일정이 나오지는
않았지만, 대략 기존 다른 프로젝트보다는 많이 걸릴것으로 협업부서에 공유를 하고는
무엇부터 해야할지 고민하기 시작했다.
(물론 이미 전부터 알고 있었던 elastic stack을 염두에 두고 있기는 했었다)

일단 Storage 시스템을 골라야 했다.

  1. Mysql
    • 주 사용 DB라서 익숙하지만,
    • 속도등의 성능면에서는 어느정도나 가능할까? 하는 의문이 들었다.
  2. Sphinx
    • 전에 검색 엔진 서비스 개발하면서 경험해봤었다.
    • 빠르긴하다.
    • 사용법도 간단하지만, 디테일한 컨트롤은 쉽지 않았다.
  3. Elastic stack
    • 경험이 전무하다.(동영상 공부 조금 해본 정도)
    • 다른 여러 많은 서비스에서 사용하고 있다.(안정적이다 라고 느낄수 있을듯)
    • 여러 방향으로 사용해 볼 수 있다.(System or application Watcher and Alert, search engine 등등)

어떻게 할 것인가?

내 맘속은 이미 “어시E”(어차피 시스템은 Elastic stack) 였다.
위의 장,단점 도 있지만 개발자 특성상 새로운 경험을 해본다는것은 매우 구미가 당기는 부분이었다.
물론 주변상황이 Elastic stack을 사용해 볼수 있는 환경이 되기도 했었다.
(상황이 안되면 도입하기가 어렵다는것을 느끼시는분들이 있을것이다 분명. ㅎㅎ)

일단 아는것이 별로 없으니 알아야 하는데, 그전에 전반적인 flow를 생각해봤다.

  1. data 수집
    1) 우리 서비스는 중요한 이벤트 마다 log를 상세히(?) 남긴다.
    2) DB를 통한 데이터 수집보다는 이게 훨씬 더 안정적이라고 생각했다.
    3) filebeats(Elastic stack의 Beats 중 하나) 를 사용하여 로그를 읽어들여서 “data 정제” 단계로 넘긴다.

    - DB io는 비용이 꽤 많이 드는것으로 안다. 반면 로그 file io는 그에 비해 싼편으로 생각했다.
- 물론 우리는 web과 db 서버가 분리되어 있어서 더욱 그렇다.
- 평상시에는 web이 db에 비해 많이 놀고 있기 때문에, 우선은 file io를 택했다.
- 이렇게 하면 운영중인 서버에 크게 부하가 없이, 통계 서비스 제공이 가능하다고 생각했다.
- Logstash 로 log 수집이 가능하나, 그에 비해 경량 data 수집기인 filebeats를 이용했다.

  2. data 정제
    1) Logstash 를 이용해서 비정형화되어 있는 data를 받는다.
    2) elasticsearch에 index mapping에 맞게 변환한다.
    3) elasticsearch에 정제된 data를 보낸다.

    - 하다 보니 Logstash로 정제하는 작업이 꽤 비용이 들것으로 예상되었다.
- Logstash의 부하를 미리 줄여보고자 애초에 log를 남기는 시점에 
  어느정도 정형화 하여 log를 남기도록 하였다.

  3. elasticsearch indexing
    1) Logstash로 부터 data를 받아서 index 생성 및 인덱싱을 한다.
    2) 통계라는것이 시간에 따른 data의 변화이기때문에(시계열 data라고들 하더라), 

    index 생성 규칙을 날짜(daily)별로 생성한다.

통계 데이터를 위한 요구 사항 정리

기본

  1. 총 매출액(=입금 완료 - 환불 금액)
  2. 주문 상태에 따른 주문 건수
  3. 방문자 수(중복 제거)
  4. 결제 금액
  5. 환불 금액

채널별

  1. 총 매출액
  2. 주문 상태에 따른 주문 건수
  3. 방문자 수(중복 제거)

결제 수단별

  1. 총 매출액
  2. 주문수

상품별 - 판매

  1. 총 판매 금액
  2. 총 판매 수량

상품별 - 환불

  1. 총 환불 금액
  2. 총 환불 수량

상품별 - 채널 인기

  1. 총 판매 금액
  2. 총 판매 수량

결론

  • 모든 index는 일별 데이터(day) 이다. ex) order-2018-10-23
  • 일별 데이터에는 그날의 주문 데이터를 넣는다.
  • 주문 데이터에는 통계에 필요한 값들을 포함한다. - 상품 통계로 인해 품목 별로 담아야 하나?
    – 상점 아이디
    – 주문 번호
    – 주문 금액
    – 주문 수량
    – 상품 번호
    – 상품 판매수
    – 채널 정보(sns 명)
    – 유입 정보(브라우저)
    – request_uri?
    – 환불 여부?
    – 일별 총 매출액, 주문수, 채널 인기 상품 등등
    – 일별 데이터에는 주문금액, 상품 정보(prd_id), 채널 정보, 유입 정보(브라우저)
